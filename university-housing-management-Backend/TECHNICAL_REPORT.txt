4. ARCHITECTURE OVERVIEW
================================================================================

  The project follows a classic layered (N-Tier) Spring Boot architecture:

  [HTTP Client]
       |
  [Controller Layer]   -- REST endpoints, receives/returns DTOs
       |
  [Service Layer]      -- Business logic, @Transactional operations
       |
  [Repository Layer]   -- Spring Data JPA, DB access
       |
  [Entity Layer]       -- JPA-mapped domain objects
       |
  [MySQL Database]

  Mapping between Entities and DTOs is handled by a single static utility class:
  EntityDtoMapper (manual mapping, no MapStruct or ModelMapper used).

  NOTE: The "ServiceImpl" folder contains interfaces (named I*Service),
        and the "Services" folder contains the actual implementations (*ServiceImpl).
        This naming is inverted from the conventional Spring naming convention.

5. DOMAIN MODEL & ENTITY RELATIONSHIPS
================================================================================

    - idUniversite (PK, auto-generated Long)
    - nomUniversite (String)
    - adresse (String)

      => Universite is the owning side of the OneToOne

  Foyer
    - idFoyer (PK, auto-generated Long)
    - nomFoyer (String)

  Bloc
    - idBloc (PK, auto-generated Long)
    - nomBloc (String)
    - capaciteBloc (Long)
    - foyer (ManyToOne -> Foyer, FK: foyer_id)
    - chambres (OneToMany -> Chambre, CascadeType.ALL, LAZY)

  Chambre
    - idChambre (PK, auto-generated Long)
    - numeroChambre (Long)
    - typeC (Enum: TypeChambre -> SIMPLE | DOUBLE | TRIPLE)
    - bloc (ManyToOne -> Bloc, FK: bloc_id)
    - reservations (OneToMany -> Reservation, CascadeType.PERSIST, LAZY)

  Etudiant
    - idEtudiant (PK, auto-generated Long)
    - nomEt (String)
    - prenomEt (String)
    - cin (Long)  -- national ID number
    - ecole (String)
    - dateNaissance (LocalDate)
    - reservations (OneToMany -> Reservation, CascadeType.PERSIST, LAZY)

  Reservation
    - idReservation (PK, String, manually generated)
      Format: "YYYY/YYYY-<nomBloc>-<numeroChambre>-<cin>"
      Example: "2025/2026-BlocA-101-12345678"
    - anneeUniversitaireDebut (LocalDate)
    - anneeUniversitaireFin (LocalDate)
    - dateReservation (LocalDate)
    - estValide (Boolean)
    - chambre (ManyToOne -> Chambre, FK: chambre_id)
    - etudiant (ManyToOne -> Etudiant, FK: etudiant_id)

  Enum: TypeChambre
    - SIMPLE  (capacity: 1)
    - DOUBLE  (capacity: 2)
    - TRIPLE  (capacity: 3)

  Relationship Summary:
    Universite  1 <---> 1  Foyer
    Foyer       1 <---> N  Bloc
    Bloc        1 <---> N  Chambre
    Chambre     1 <---> N  Reservation
    Etudiant    1 <---> N  Reservation

10. MAPPER  (EntityDtoMapper.java)
================================================================================

  Static utility class with no Spring injection.
  Provides bidirectional conversion for all 6 domain objects.

  Methods:
    toDto(Universite)   -> UniversiteDTO   (includes nested FoyerDTO)
    toEntity(UniversiteDTO) -> Universite

    toDto(Foyer)        -> FoyerDTO        (includes nested List<BlocDTO>)
    toEntity(FoyerDTO)  -> Foyer

    toDto(Bloc)         -> BlocDTO         (includes nested List<ChambreDTO>)
    toEntity(BlocDTO)   -> Bloc

    toDto(Chambre)      -> ChambreDTO      (no reservations to avoid circular load)
    toEntity(ChambreDTO) -> Chambre

    toDto(Etudiant)     -> EtudiantDTO
    toEntity(EtudiantDTO) -> Etudiant

    toDto(Reservation)  -> ReservationDTO  (includes idChambre, idEtudiant as flat IDs)
    toEntity(ReservationDTO) -> Reservation (Chambre/Etudiant must be set separately in service)

  Note: Chambre and Reservation associations are intentionally excluded from
        ChambreDTO and ReservationDTO to prevent circular serialization issues.

11. OBSERVATIONS & NOTES
================================================================================

  1. NAMING INCONSISTENCY:
     - The folder "ServiceImpl" contains interfaces (I*Service), while "Services"
       contains implementations (*ServiceImpl). This is the reverse of the
       conventional Spring Boot naming convention.
     - Class "chambreController" starts with a lowercase letter, violating
       Java naming conventions (should be "ChambreController").

  2. NO SECURITY LAYER:
     - No Spring Security, JWT, or authentication mechanism is present.
     - All endpoints are publicly accessible.

  3. NO CORS CONFIGURATION:
     - No @CrossOrigin or global CORS config found. Cross-origin requests from
       a frontend may be blocked by the browser.

  4. NO GLOBAL EXCEPTION HANDLER:
     - No @ControllerAdvice or @RestControllerAdvice for centralized error handling.
     - RuntimeExceptions thrown in services will result in unformatted 500 responses.

  5. NO PAGINATION:
     - All "getAll" endpoints return full lists with no pagination or filtering.
     - Could be a performance concern with large datasets.

  6. NO API DOCUMENTATION:
     - No Swagger/OpenAPI (springdoc-openapi) dependency configured.

  7. RESERVATION ID GENERATION:
     - The composite ID logic in ReservationServiceImpl uses string concatenation
       with year % 100, which may produce incorrect results for years like 2100.

  8. EMPTY PASSWORD IN CONFIG:
     - application.yaml has an empty MySQL password. This is fine for local dev
       but must be secured for any other environment.

  9. NO VALIDATION:
     - No @Valid / @NotNull / @Size annotations on DTOs or controller parameters.
     - Input is not validated before hitting the service layer.

  10. FOYER <-> UNIVERSITE BIDIRECTIONAL SYNC:
      - The bidirectional OneToOne between Foyer and Universite requires manual
        sync in service methods (foyer.setUniversite / universite.setFoyer).
        This is correctly handled in FoyerServiceImpl and UniversiteServiceImpl.
